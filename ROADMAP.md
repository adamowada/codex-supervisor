# Roadmap

`ROADMAP.md` is the stable master build plan for `codex-supervisor`. It is written for new Codex
sessions that need to understand the intended product, the order of major capabilities, and the
quality bars for each stage.

It is not the live work selector and it is not a changelog. Do not edit this file to record that a
stage moved forward. Live queue and progress values belong only in `plans/planning.sqlite3`; the
human-readable resume point belongs in `HANDOFF.md`.

Change this file only when the master plan itself changes. After an intentional protected-doc edit,
use the `source-lock-operator` skill and refresh the protected hashes.

## Operating Loop For New Codex Sessions

Every session working in this repo follows the same control flow:

1. Use the `codex-supervisor` meta skill first.
2. Inspect `story-loop-status --json` before choosing work.
3. Inspect the selected task with `task-show <task_id> --json` or `task-current --json`, depending
   on the queue state.
4. Read only the source-of-truth files, insights, skills, and source snippets required by the task.
5. Render or confirm a Goal Contract for the selected vertical slice.
6. Use the smallest applicable skill, routed by `skill-router` when needed.
7. Make narrowly scoped changes in allowed paths.
8. Run the verification commands named by the task, then the broader gate required by `TESTING.md`.
9. Run review with `fresh-thread-code-reviewer` when the task touches production behavior,
   contracts, architecture, public surfaces, or source-of-truth docs.
10. Fix accepted findings with `review-finding-fixer`.
11. Update `plans/planning.sqlite3` and `HANDOFF.md` as the only live progress surfaces.
12. Update `insights/` with `knowledge-graph-updater` when a durable workflow lesson appears.
13. Publish with `acp-publisher` only after verification, source locks, public hygiene, and git
    state are clean enough for the requested checkpoint.

## Global Invariants

- `plans/planning.sqlite3` owns operational queue state.
- `HANDOFF.md` owns the mutable resume summary.
- Protected source-of-truth docs define doctrine and contracts, not task progress.
- `sources/` clones are ignored study material, not vendored source.
- Workers operate on one vertical slice in one isolated context.
- AFK tasks require acceptance criteria, verification commands, allowed paths, and stop conditions.
- HITL tasks require a human decision before unattended execution continues.
- Codex local SQLite databases are read-only telemetry.
- Native Codex Goals are execution aids, not the project queue.
- MCP and plugins are interfaces, not orchestration state owners.
- Review and deterministic verification gate production-intended changes.
- Repeated corrections become insights or skills.

## Stage 0: Bootstrap Repository

### Purpose

Establish the repository as a public-safe, Python-first supervisor project with durable source of
truth, ignored study sources, source locks, a planning database, verification scripts, skills, and a
clean handoff path.

### Inputs

- Human goals for an agentic engineering factory.
- Existing project patterns from `nlp-stock-prediction`, `codex-subagent-testing`,
  `observe-safety-monorepo`, and `tech-resume`.
- OSS inspiration sources listed in `sources/README.md`.

### Procedure

1. Create the repository shell and initialize git.
2. Add protected source-of-truth documents.
3. Add MIT licensing and attribution policy.
4. Add ignored `sources/` clones and inventory checks.
5. Add Python package skeleton, CLI entry point, and verification scripts.
6. Add planning SQLite seed state and migration helpers.
7. Add repo-local skills for source locks, planning, story loops, review, ACP, and learning.
8. Add public hygiene checks so source clones, secrets, local paths, and generated artifacts remain
   out of commits.
9. Record handoff instructions for fresh Codex sessions.

### Skills

- `source-evidence-attributor`
- `source-lock-operator`
- `setup-agent-docs`
- `spawned-project-bootstrap`
- `acp-publisher`

### Done When

- Protected source-of-truth documents exist and pass the lock guard.
- Public hygiene and file-purpose checks pass.
- Ignored study sources are reproducible from `sources/README.md`.
- Planning SQLite opens through typed helpers.
- The default verification gate passes.
- ACP records the bootstrap checkpoint.

## Stage 1: Planning SQLite Core

### Purpose

Make tracked SQLite the durable operational store for plans, milestones, acceptance criteria,
decisions, progress events, artifact links, commit links, supervisor tasks, and worker runs.

### Inputs

- `PLANS.md`
- `CONTRACTS.md`
- Existing queue needs from the supervisor workflow.

### Procedure

1. Define schema migrations with explicit version checks.
2. Add typed records and serializers for every table.
3. Add idempotent initialization and migration entry points.
4. Add CLI commands for plan, task, worker-run, artifact, progress, decision, and commit flows.
5. Keep orientation commands read-only.
6. Make task claiming atomic so two workers cannot claim the same AFK slice.
7. Add integrity checks for terminal parent rows, open child work, worker-result artifacts, and
   acceptance evidence.
8. Add tests for schema drift, lifecycle transitions, and CLI behavior.

### Skills

- `planning-sqlite-operator`
- `goal-contract-drafter`
- `story-loop-runner`

### Done When

- `plan-init` initializes or migrates only when explicitly invoked.
- Orientation commands inspect existing state without mutation.
- `story-loop-status` is the queue state machine.
- `task-current` selects only executable AFK work.
- Worker-run completion requires durable result evidence.
- Planning integrity checks are part of the default gate.

## Stage 2: Source Locks And Public Hygiene

### Purpose

Protect stable doctrine, publication safety, and repo cleanliness while still allowing intentional
source-of-truth changes.

### Inputs

- Protected file list in `AGENTS.md`.
- Public repo policy in `ATTRIBUTIONS.md` and `sources/README.md`.
- Git index state.

### Procedure

1. Hash protected source-of-truth files.
2. Verify hashes, git tracking, line endings, and binary handling.
3. Verify every public file has a declared purpose.
4. Verify ignored study sources remain unstaged.
5. Verify public hygiene for local roots, secrets, generated artifacts, and cloned source trees.
6. Add a deliberate re-lock path through `source-lock-operator`.
7. Require the publication gate before ACP and release checkpoints.

### Skills

- `source-lock-operator`
- `source-evidence-attributor`
- `acp-publisher`

### Done When

- Protected-file hash failures are actionable and narrow.
- Intentional protected-doc edits have a repeatable hash refresh path.
- Publication-ready verification fails on untracked or unstaged public files.
- Ignored source clones, worker runs, worktrees, logs, and local caches remain out of commits.

## Stage 4: Task Compiler And Goal Contract Assembly

### Purpose

Turn plans and project facts into small executable vertical slices with enough structure for
fresh-context workers.

### Inputs

- Planning rows.
- Project adapter facts.
- Stable source-of-truth docs.
- Human planning-session decisions.

### Procedure

1. Compile milestones and acceptance criteria into candidate tasks.
2. Classify each task as `AFK` or `HITL`.
3. Attach dependencies, allowed paths, out-of-scope boundaries, and verification commands.
4. Detect vague or horizontal tasks and send them through `factory-task-decomposer`.
5. Render Goal Contracts with objective, context, boundaries, verification, stop condition, blocked
   condition, iteration policy, and record-update expectations.
6. Store task contracts in planning SQLite.
7. Add tests for AFK readiness, dependency sorting, and prompt/contract rendering.

### Skills

- `factory-task-decomposer`
- `afk-issue-shaper`
- `goal-contract-drafter`
- `verification-command-picker`
- `grill-with-docs`

### Done When

- Every ready AFK task has nonempty acceptance criteria, verification commands, and allowed paths.
- HITL tasks explain the exact human decision required.
- Goal Contract rendering works with native Goal Mode and prompt-rendered fallback.
- The compiler rejects broad "build everything" work without vertical-slice decomposition.

## Stage 5: Story Loop Orchestrator

### Purpose

Run the factory as a bounded loop: one ready AFK task, one fresh context, one verification surface,
one review path, one durable record.

### Inputs

- Ready supervisor tasks.
- Goal Contracts.
- Worker backend configuration.
- Verification and review policy.

### Procedure

1. Select the highest-priority executable AFK slice from current-queue plans.
2. Stop on running, HITL, blocked, empty, or policy-gated queue states.
3. Prepare worker metadata and an isolated workspace.
4. Execute exactly one story.
5. Capture structured result evidence.
6. Run deterministic verification.
7. Run automated review when required.
8. Create repair tasks for accepted findings or mark blockers with evidence.
9. Record progress events, artifacts, insight updates, and handoff notes.
10. Repeat only when policy allows another ready AFK slice.

### Skills

- `story-loop-runner`
- `fresh-context-worker`
- `worktree-sandbox-guard`
- `fresh-thread-code-reviewer`
- `review-finding-fixer`
- `context-compaction-handoff`

### Done When

- Queue states are explicit and test covered.
- One iteration cannot silently broaden into unrelated work.
- Worker results, verification, review, and repair decisions are recorded.
- A fresh Codex thread can resume from planning SQLite plus `HANDOFF.md`.

## Stage 6: Codex Exec Worker Backend

### Purpose

Launch fresh-context Codex workers noninteractively with controlled prompts, isolated worktrees,
structured output schemas, captured evidence, and safe result ingestion.

### Inputs

- Goal Contract.
- Task row and allowed paths.
- Project adapter verification commands.
- Codex executable and `CODEX_HOME` policy.
- Worker result JSON schema.

### Procedure

1. Resolve the Codex executable and record `codex --version`.
2. Resolve the intended `CODEX_HOME`, config path, sandbox settings, and Goal Mode feature state.
3. Decide whether native Goals are available for the selected worker path.
4. Render prompt fallback when native Goals are unavailable.
5. Create a disposable worktree with the correct branch base.
6. Launch `codex exec --json --output-schema` with explicit working directory, prompt, sandbox,
   approval policy, and output paths.
7. Capture JSONL events, stdout, stderr, exit code, duration, diff, and final structured result.
8. Validate worker output against the Worker Result Contract.
9. Update `worker_runs`, artifact links, progress events, and queue state transactionally.
10. Preserve raw evidence while keeping ignored run output out of publication commits.

### Skills

- `fresh-context-worker`
- `worktree-sandbox-guard`
- `goal-contract-drafter`
- `windows-shell-quoting`
- `subagent-run-digest`

### Done When

- A fake backend and Codex Exec backend share the same result-ingestion contract.
- Failed launches record failure class, logs, and retry guidance.
- Completed worker runs cannot advance without valid JSON result evidence.
- Worker prompts remain bounded and source-linked.
- Windows and Linux command quoting are covered by tests.

### Skills

## Stage 8: Review And Verification Loop

### Purpose

Turn worker output into production-quality code by combining deterministic checks, code review,
architecture review, and focused repair loops.

### Inputs

- Worker diff.
- Task contract.
- Source-of-truth docs.
- Verification logs.
- Worker result evidence.

### Procedure

1. Run task verification commands first.
2. Run the default gate from `TESTING.md` when the task risk profile requires it.
3. Launch review lanes for the requested scope: entire repo, branch, commit, or diff.
4. Route review mode: everything, code quality, architecture, or source-of-truth drift.
5. Use `improve-codebase-architecture` for architecture-only reviews.
6. Classify findings by severity, evidence, and whether HITL disagrees.
7. Fix accepted findings with focused tasks.
8. Re-run verification after fixes.
9. Record review artifacts, waivers, and follow-up tasks.

### Skills

- `fresh-thread-code-reviewer`
- `improve-codebase-architecture`
- `review-finding-fixer`
- `diagnose`
- `tdd`
- `ci-repair-loop`

### Done When

- Review modes are explicit and reproducible from a fresh thread.
- Accepted findings become repair work or direct fixes.
- Waived findings have a recorded rationale.
- Verification evidence matches the task acceptance criteria.
