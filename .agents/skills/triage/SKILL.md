# Triage

Move issues through a small state machine of triage roles while preserving the repo's source-of-truth order.

Every comment or issue posted to a public issue tracker during triage **must** start with this disclaimer:

```markdown
> *This was generated by AI during triage.*
```

## Reference Docs

- [AGENT-BRIEF.md](AGENT-BRIEF.md) - how to write durable agent briefs.
- [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md) - how the `.out-of-scope/` knowledge base works.

If the issue tracker, role labels, or source-of-truth order are missing, run `setup-agent-docs` before changing issue state.

## Source Order

For `codex-supervisor` spawned projects, use this order unless the repo documents a different one:

1. Locked source-of-truth docs.
2. Planning SQLite.
3. GitHub issues and PRs.
4. Handoff artifacts.
5. Chat/session history.

When these sources conflict, surface the conflict before writing comments, changing labels, or closing anything.

## Roles

Two **category** roles:

- `bug` - something is broken.
- `enhancement` - new feature or improvement.

Five **state** roles:

- `needs-triage` - maintainer needs to evaluate.
- `needs-info` - waiting on reporter or HITL.
- `ready-for-agent` - fully specified, ready for an AFK agent.
- `ready-for-human` - needs human implementation.
- `wontfix` - will not be actioned.

Every triaged issue should carry exactly one category role and one state role. If state roles conflict, flag it and ask the maintainer before doing anything else.

The actual label strings used by the issue tracker may differ. Use the mapping in `docs/agents/triage-labels.md` or the repo's equivalent setup docs.

## Invocation

The user describes what they want in natural language. Interpret the request and act. Examples:

- "Show me anything that needs my attention."
- "Let's look at #42."
- "Move #42 to ready-for-agent."
- "What's ready for agents to pick up?"

## Show What Needs Attention

Query the configured issue tracker and present three buckets, oldest first:

1. **Unlabeled** - never triaged.
2. **`needs-triage`** - evaluation in progress.
3. **`needs-info` with reporter activity since the last triage notes** - needs re-evaluation.

Use GitHub connector tools when available. Use `gh` CLI only when the connector is unavailable and the repo already uses GitHub CLI workflows.

Show counts and a one-line summary per issue. Let the maintainer pick unless the user asked for full-auto triage of a specific bounded set.

## Triage A Specific Issue

1. **Gather context.** Read the full issue: body, comments, labels, reporter, dates, linked PRs, and prior triage notes. Read planning SQLite records or source docs linked from the issue. Explore the codebase using the project's domain glossary and ADRs. Read `.out-of-scope/*.md` and surface prior rejections that resemble this issue.

2. **Recommend.** Tell the maintainer your category and state recommendation with reasoning, plus a brief codebase summary relevant to the issue. In full-auto mode, proceed when the evidence is strong and the action is reversible; otherwise pause for HITL.

3. **Reproduce bugs.** Before any grilling, attempt reproduction: read the reporter's steps, trace the relevant code, and run tests or commands. Report whether reproduction succeeded, failed, or needs more information. A confirmed repro makes a stronger agent brief.

4. **Grill if needed.** If the issue needs fleshing out, run `grill-with-docs`.

5. **Apply the outcome:**
   - `ready-for-agent`: post an agent brief comment and create or update the linked planning SQLite task when that system is active.
   - `ready-for-human`: use the same structure as an agent brief, but state why it cannot be delegated yet.
   - `needs-info`: post triage notes using the template below.
   - `wontfix` bug: post a polite explanation, then close if the maintainer has authorized that outcome.
   - `wontfix` enhancement: write or update `.out-of-scope/`, link to it from a comment, then close if authorized.
   - `needs-triage`: apply the role. Add an optional progress comment if useful.

## Quick State Override

If the maintainer says "move #42 to ready-for-agent", trust them and apply the role directly. Confirm the exact role changes, comments, and close/reopen actions before acting unless the user has explicitly asked for full-auto operation. Skip grilling. If moving to `ready-for-agent` without a grilling session, ask whether they want an agent brief or create one from the available evidence.

## Needs-Info Template

```markdown
## Triage Notes

**What we've established so far:**

- point 1
- point 2

**What we still need from you (@reporter):**

- question 1
- question 2
```

Capture everything resolved during grilling under "established so far" so the work is not lost. Questions must be specific and actionable, not "please provide more info."

## Resuming A Previous Session

If prior triage notes exist, read them, check whether the reporter has answered any outstanding questions, and present an updated picture before continuing. Do not re-ask resolved questions.
